- Status: DRAFT
- Created: 2026-04-12
- Owners: lotus-advise
- Requires Approval From: lotus-advise, lotus-core, lotus-risk, lotus-manage maintainers
- Depends On: RFC-0006, RFC-0010, RFC-0012, RFC-0013, RFC-0015, RFC-0019, RFC-0020

## Executive Summary

`lotus-advise` has a strong proposal execution backbone: canonical simulation from `lotus-core`, concentration risk from `lotus-risk`, persisted lifecycle, replay evidence, workspace handoff, async operations, and delivery evidence. The next enterprise-grade gap is advisory decision quality.

A private banker should not have to infer the business answer from raw allocation, risk, suitability, and workflow payloads. Every proposal should produce one deterministic, evidence-backed decision summary that answers:

1. Is the proposal suitable for this client and mandate?
2. What blocks the proposal or requires review?
3. What changed materially across allocation, risk, liquidity, currency, cash, product exposure, and suitability?
4. Which approvals are required and why?
5. What is the next best operational action for the advisor?
6. Which evidence supports each conclusion?

This RFC defines an in-place enhancement to the existing proposal contract. Because `lotus-advise` is not live and all callers are controlled, this RFC does not introduce public `/v2` routes, `advisory-simulation.v2`, or duplicate API families. It strengthens existing proposal APIs as pre-live hardening. Future post-live breaking changes require a dedicated versioning RFC.

The target outcome is a first-class `proposal_decision_summary` generated by an Enterprise Suitability Policy and Decision Engine. It is deterministic, auditable, replayable, and safe for UI, artifact, reporting, and future AI narrative consumers.

## Why This RFC Exists

Recent implementation work materially improved the platform foundation:

1. RFC-0020 aligned proposal allocation with canonical `lotus-core` authority.
2. `lotus-risk` concentration lens is integrated and validated against direct upstream analytics.
3. Proposal lifecycle, versioning, async operations, replay, delivery, and workspace handoff are strong.
4. Live validation covers no-op, changed-state, cross-currency, non-held instrument, degraded runtime, and async lifecycle paths.

The remaining product problem is that a technically correct proposal is not the same as a private-banking-grade advisory recommendation.

A relationship manager, compliance reviewer, risk officer, and client reviewer need a decision summary that is:

1. exception-first,
2. business-language-oriented,
3. evidence-backed,
4. consistent across APIs and UI,
5. replayable for audit,
6. grounded in domain-authoritative services,
7. strong enough to support later AI-generated narrative without letting AI invent business conclusions.

Without this RFC, each UI or artifact consumer will be tempted to reconstruct decision logic locally. That would create duplicated business rules, inconsistent wording, weak auditability, and superficial UI features not backed by real backend functionality.

## Current State

### What Is Already Strong

`lotus-advise` already has these strengths:

1. `POST /advisory/proposals/simulate` supports stateless and stateful proposal inputs.
2. Stateful context resolution uses `lotus-core` as portfolio authority.
3. Proposal results include before/after simulated state, allocation views, risk lens, suitability output, rule results, gate decision, lineage, and replay evidence.
4. Proposal lifecycle supports create, version, approvals, state transitions, execution handoff, execution updates, report requests, delivery summary, and delivery events.
5. Async create/version operations are idempotent and replayable.
6. Workspace evaluate/save/handoff flows preserve stateful context and risk evidence.
7. Live validation proves canonical cross-service parity.

### What Is Not Yet Business Grade

The current proposal surfaces still leave advisory decision synthesis too distributed:

1. Top-level statuses `READY`, `PENDING_REVIEW`, and `BLOCKED` are necessary but not sufficiently explanatory.
2. Suitability output is useful but not yet a full enterprise suitability policy model.
3. Client, mandate, jurisdiction, product eligibility, and advisory agreement context are not first-class decision inputs.
4. Material-change classification is not a stable backend contract.
5. Approval requirements are not represented as one consolidated advisor-facing decision object.
6. Missing evidence is not consistently separated from actual pass/fail outcomes.
7. Future AI narrative work lacks a deterministic evidence spine.

## Problem Statement

`lotus-advise` needs to become the advisory decision orchestration authority without becoming the source of truth for portfolio, allocation, or risk calculations.

Today, proposal consumers can see many facts, but not one canonical decision object. That creates these risks:

1. UI duplicates private-banking decision rules.
2. Artifacts summarize proposal state inconsistently.
3. Compliance and risk reviewers receive incomplete or uneven evidence.
4. Client-facing summaries may overstate readiness or understate blockers.
5. AI narratives, if added later, may be grounded in the wrong layer.
6. Proposal replay may preserve raw evidence but not the advisor decision posture that existed when the version was created.

A banking-grade platform must make the decision layer deterministic and explicit.

## Goals

1. Add a first-class `proposal_decision_summary` to existing proposal surfaces.
2. Strengthen the existing suitability capability into an Enterprise Suitability Policy and Decision Engine.
3. Preserve existing public API families and enhance them in place.
4. Use internal policy/evidence versions where useful, without introducing public API v2 contracts.
5. Model client, mandate, jurisdiction, product, portfolio, allocation, risk, liquidity, and evidence completeness as decision inputs.
6. Classify material changes across the proposal before/after state.
7. Produce deterministic approval requirements and next actions.
8. Persist decision evidence with proposal versions, workspace saved versions, async replay, and proposal artifacts.
9. Make the decision summary directly consumable by UI and reporting without duplicated frontend logic.
10. Create a safe deterministic foundation for future AI narrative and proposal alternative work.

## Non-Goals

1. Do not introduce `/v2` route families.
2. Do not introduce `advisory-simulation.v2`.
3. Do not replace `lotus-core` as simulation, AUM, valuation, or allocation authority.
4. Do not replace `lotus-risk` as risk calculation authority.
5. Do not implement proposal alternative generation. That belongs in a later RFC.
6. Do not implement AI narratives. That belongs in a later RFC.
7. Do not implement every jurisdiction policy pack in one delivery slice.
8. Do not allow UI code to own suitability, approval, or decision-routing rules.
9. Do not create client-facing output without explicit review and artifact controls.

## Pre-Live Contract Hardening Decision

Because `lotus-advise` is not live and all callers are controlled, this RFC updates existing proposal contracts in place.

Allowed changes:

1. Additive fields on existing response models.
2. Stronger semantics for existing suitability, gate, artifact, replay, and lifecycle evidence.
3. Internal policy version fields such as `decision_policy_version` and `suitability_policy_version`.
4. Internal evidence schema names where needed for audit.
5. New read-only projection endpoints under the existing `/advisory/proposals/...` route family.

Disallowed changes:

1. Public `/v2` route families.
2. Separate proposal simulation version families.
3. Duplicate DTOs that model the same domain under different names.
4. Compatibility scaffolding for external clients that do not exist yet.

Future rule:

1. after live adoption begins, breaking external contract changes require a dedicated versioning RFC,
2. additive pre-live hardening can continue only while all callers remain controlled,
3. every new API field must be backed by real backend evidence, not UI-only inference.

## Domain Authority Boundaries

### lotus-core

`lotus-core` remains the authority for:

1. portfolio state,
2. positions,
3. cash,
4. AUM,
5. valuation,
6. allocation,
7. reference data,
8. market data,
9. proposal simulation execution.

`lotus-advise` must not recalculate these for production behavior.

### lotus-risk

`lotus-risk` remains the authority for:

1. concentration analytics,
2. future risk lenses such as volatility, VaR, expected shortfall, liquidity risk, credit risk, stress, duration, and factor exposure when implemented,
3. risk methodology and risk calculation lineage.

`lotus-advise` may classify risk evidence into advisory decisions, but it must not calculate risk itself.

### lotus-manage or Client/Mandate Authority

Client and mandate context should come from the canonical client/portfolio management authority. `lotus-advise` should consume:

1. client risk profile,
2. mandate type,
3. advisory agreement,
4. jurisdiction and booking center,
5. knowledge and experience,
6. investment objective,
7. time horizon,
8. liquidity needs,
9. restrictions and preferences,
10. ESG or exclusion preferences where available.

If the canonical upstream is unavailable, the decision summary must degrade explicitly rather than pretending suitability is complete.

### lotus-advise

`lotus-advise` owns:

1. advisory suitability policy application,
2. proposal decision summary,
3. approval requirement classification,
4. proposal next-action classification,
5. proposal lifecycle and workspace evidence,
6. persistence and replay of advisory decision evidence.

## Target Capability

The RFC introduces an Enterprise Suitability Policy and Decision Engine composed of four internal layers:

1. evidence collection,
2. suitability policy evaluation,
3. material-change classification,
4. decision-summary projection.

The layers must remain modular. Each layer should be testable independently and reusable across simulate, create, version, workspace, artifact, and replay paths.

## Target API Contract

### Additive Response Field

Add `proposal_decision_summary` to existing proposal outputs where proposal evidence is available:

1. `POST /advisory/proposals/simulate`,
2. `POST /advisory/proposals`,
3. `POST /advisory/proposals/{proposal_id}/versions`,
4. async operation replay after success,
5. workspace evaluate/save/handoff,
6. artifact generation,
7. proposal detail/version detail,
8. replay evidence.

### Optional Projection Endpoint

Add a read-only endpoint if the UI needs to refresh or inspect persisted decision state directly:

1. `GET /advisory/proposals/{proposal_id}/decision-summary`

This endpoint must project from persisted proposal version evidence. It must not call upstream services or silently recompute with newer market data.

## Architecture Direction

### Decision Evidence Collector

A reusable evidence collector should assemble a normalized internal input object from:

1. canonical simulation result,
2. risk lens,
3. current suitability output,
4. lifecycle state,
5. resolved context,
6. client context,
7. mandate context,
8. product and shelf metadata,
9. missing-evidence diagnostics.

The collector should not make policy decisions. It only normalizes evidence.

### Suitability Policy Engine

The policy engine should apply deterministic policy packs to the evidence object.

Required properties:

1. pure functions where possible,
2. Decimal-safe calculations,
3. explicit policy version,
4. stable reason codes,
5. no HTTP calls inside policy evaluation,
6. no UI-specific behavior.

### Material Change Classifier

The classifier should compare before/after evidence and produce business-readable deltas.

It should be configurable by policy thresholds, not hardcoded inside routes.

### Decision Summary Projector

The projector converts policy and material-change evidence into `proposal_decision_summary`.

It should define deterministic precedence:

1. blocking data quality and missing required evidence,
2. hard rule failures,
3. blocking suitability failures,
4. risk review requirements,
5. compliance review requirements,
6. client consent requirements,
7. revision recommendations,
8. ready for client review.

## Implementation Discipline

Implementation must follow these rules:

1. Keep models small and composable; do not create one giant suitability object that mixes evidence collection, policy evaluation, and presentation.
2. Keep policy functions deterministic and side-effect free.
3. Use Decimal-safe calculation for monetary, allocation, weight, and threshold logic.
4. Put route handlers behind services; do not implement decision logic in API routes.
5. Keep persistence append-only where decision evidence is part of proposal version history.
6. Add helper modules only when they reduce duplication across simulate, lifecycle, workspace, artifact, and replay paths.
7. Prefer explicit reason codes over free-form text when behavior affects workflow routing.
8. Write tests that prove business outcomes, not only schema shape.
9. Preserve small commits by slice: models/projector, persistence/replay, policy engine, context integration, material changes, live validation, documentation/branch hygiene.

## Persistence and Replay

Persist the decision summary with every proposal version.

Required behavior:

1. simulation returns a summary for transient response consumers,
2. proposal create/version persists the summary,
3. async replay returns the persisted summary after success,
4. workspace saved versions preserve the summary,
5. workspace handoff preserves the summary into lifecycle evidence,
6. proposal replay returns the exact persisted summary,
7. new proposal versions generate fresh decision summaries and do not inherit stale approvals or decision posture.

## Degraded Behavior

Decision summaries must distinguish actual failure from insufficient evidence.

Examples:

1. missing `lotus-risk` evidence means risk posture is unavailable, not passed,
2. missing client profile means client suitability is insufficiently evidenced,
3. missing mandate context means mandate alignment is unknown,
4. missing product complexity means product suitability is uncertain,
5. missing market data remains blocking when simulation cannot complete.

The decision summary must never make a positive suitability or risk claim from missing evidence.

## API and UI Alignment

UI must consume backend decision fields rather than reconstructing private-banking policy logic.

UI expectations:

1. summary first,
2. exceptions before raw data,
3. material changes before detailed tables,
4. decision badges derived from backend fields,
5. drill-down links to evidence refs,
6. no client-ready copy generated locally from raw fields.

If the UI needs a new widget, the backend must expose supported data first.

## Documentation and Agent Context Principle

This RFC intentionally includes a final slice for documentation, agent context, and branch hygiene.

The reason is operational: new repeatable advisory patterns should become reusable context for future agents and engineers. If implementation reveals a durable pattern, that pattern should be recorded in the repo-local engineering context or the platform agent context rather than rediscovered in later chats.

## Risks and Mitigations

### Risk: Decision summary becomes a second calculation engine

Mitigation:

1. enforce domain authority boundaries,
2. decision summary references canonical calculation evidence,
3. no allocation/risk/performance recalculation in `lotus-advise`.

### Risk: Suitability policy becomes too monolithic

Mitigation:

1. use policy packs,
2. keep dimensions modular,
3. test each dimension independently,
4. avoid route-local rules.

### Risk: UI still duplicates decision logic

Mitigation:

1. expose complete backend summary,
2. document UI expectations,
3. add contract tests for supported fields,
4. reject superficial UI-only decision features.

### Risk: Missing client/mandate context blocks too much too early

Mitigation:

1. support shadow-mode rollout,
2. degrade explicitly,
3. distinguish insufficient evidence from unsuitability,
4. add policy controls for required versus optional context.

## Completion Criteria

This RFC is implemented when:

1. every proposal simulation emits a decision summary or truthful degraded summary,
2. persisted proposal versions store and replay decision summaries,
3. workspace save/handoff and async replay preserve decision evidence,
4. enterprise suitability policy evaluates the required dimensions where evidence exists,
5. missing evidence is explicit and cannot appear as a pass,
6. material changes and approval requirements are deterministic,
7. artifacts and UI-ready responses use the same backend-generated summary,
8. live validation proves ready, blocked, review, and insufficient-evidence paths,
9. documentation, agent context assessment, and branch hygiene are completed in the final slice.
